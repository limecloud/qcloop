# qcloop v0.4.0

发布日期：2026-05-10

## 亮点

- 修复批次终态语义：只有全部 item 成功时才显示 `completed / 全部通过`；存在 `failed` 或 `exhausted` item 时，批次终态统一为 `failed / 未全部通过`。
- 修复历史批次状态漂移：读取批次详情或列表时会自动收敛旧数据，避免“有已耗尽项但仍显示已完成”的误导。
- 增强 verifier 证据输入：verifier prompt 现在可使用 `{{stdout}}`、`{{stderr}}`、`{{exit_code}}`、`{{attempt_status}}`、`{{attempt_type}}`。
- 增强 repair 闭环：repair 轮次会拿到上一轮 stdout / stderr / exit code 与 verifier feedback，用于直接修复并复测。
- Web 面板补充已耗尽解释：当批次存在失败或已耗尽项时，详情页会提示“未全部通过”，并说明 `max_qc_rounds=1` 不会进入 repair。

## 修复与体验改进

- 队列路径和同步 runner 路径统一使用同一套终态计算。
- `GET /api/jobs/:id` 与 `GET /api/jobs` 会修正旧批次的终态漂移。
- 批次状态文案从“已完成 / 失败”调整为“全部通过 / 未全部通过”，降低误解。
- 创建 / 编辑批次时提示推荐质检轮次：需要自动修复时建议 `3-5` 轮。
- `llms.txt`、`llms-full.txt` 与 README 同步说明状态语义、repair 输入和 verifier 占位符。
- 修正 `.gitignore` 中根二进制忽略规则，避免误忽略 `cmd/qcloop` CLI 源码入口。

## 验证

- `go test -timeout 120s ./...`
- `cd web && npm run build`
- 临时端口 API smoke：构造旧数据 `completed + exhausted item`，验证 `GET /api/jobs/:id` 与 `GET /api/jobs` 均自动收敛为 `failed`。

## 已知限制

- `exhausted` 仍然表示达到最大质检轮次或预算后未通过；如果希望 repair 自动介入，需要把 `max_qc_rounds` 设置为 `2` 以上，推荐 `3-5`。
- 本版本不改变旧 attempt / qc_round 历史记录，只修正批次级终态展示和后续运行语义。
